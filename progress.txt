# RALPH Progress Log

## Codebase Patterns

- **EntityMap is panel-only.** `src/store.ts` exposes `getEntityMap` / `setEntityMap` for the panel's Settings tab. The two cards (`hm-thermostat-card`, `hm-savings-card`) take their entity wiring from Lovelace card config (`HmThermostatCardConfig.entities` / `HmSavingsCardConfig.entities`), NOT from the panel-level EntityMap. Don't conflate them when refactoring entity selection.
- **Legacy entity-map migration runs on read.** `getEntityMap()` filters out legacy keys (`indoor_temp`, `outdoor_temp`, `power`) and rewrites the localStorage entry when any legacy key is present. Future shape changes should follow the same one-shot read-side migration pattern rather than versioning.
- **Settings entity selects are domain-filtered.** Panel filters `Object.keys(hass.states)` by `id.startsWith('climate.')` for the climate field and `'weather.'` for the weather field. Tests must include at least one entity in each target domain plus a non-matching one (e.g. `light.kitchen`) to exercise filter behavior.
- **Test fixtures need attributes for new shape.** `climate.*` entities expose `current_temperature` and `temperature` (target) under `attributes`; `weather.*` exposes `temperature` (current outdoor) under `attributes`. Tests that simulate hass states should include those attributes so future card UI changes can read them without test churn.
- **Settings persistence is draft-then-save.** Entity map and pricing zone live in two layers: `_entityMap`/`_auth.user.pricing_location` are the saved baselines; `_entityMapDraft`/`_zoneDraft` are the in-memory drafts the form binds to. Change handlers only mutate drafts. The Save button is the single persistence path (calls `setEntityMap` and `patchMe`); Reset snaps drafts back to baselines; both buttons disable when `_isDirty()` is false.
- **`<select>` value binding for happy-dom remounts.** Lit's `.value=${selected}` does not reliably reflect on a select after a remount in happy-dom. Use `?selected=${id === selected}` per `<option>` (combined with the `.value` binding for the change-event path) so case-E-style "remount restores the saved value" tests pass without requiring real-browser semantics.
- **Display labels for backend lookups belong in `src/data/`.** When the backend exposes integer keys (pricing zones, appliance types) and the UI needs human-readable labels, drop the mapping in `src/data/<topic>.ts` as a typed `Record<KeyUnion, Label>` plus pure label-builder helpers. Keep helpers tolerant of out-of-range keys (fallback to a TBD label) so callers don't have to clamp. Drive `<option>` lists off `Object.keys(MAP).map(Number).sort()` instead of a hardcoded tuple — adding/removing a zone becomes a one-line data edit.
- **Wrap fragment-level `${TemplateResult}` interpolations in a host element.** When a Lit template has `${someTemplateResult}` at fragment level between sibling elements (e.g. `</label>\n      ${typeFields}\n      <div>`), the parser under happy-dom silently drops the binding slot. All bindings AFTER it shift by one position, leading to surreal output like a button's text rendering its own `@click` handler source. Wrap the slot in `<div>${typeFields}</div>` (or any host element) and the parser handles it correctly. This was a multi-hour debug rabbit hole on US-FE-APPLIANCES-01; future stories that compose multi-section templates should default to wrapping each section in its own element.
- **API `create` shape is normalize-on-the-way-in.** `appliancesApi.create()` accepts either a full `Appliance` body or a thin `{appliance_id}` envelope from the server and synthesizes an `Appliance`-shaped object from the input + response. Callers always get the same shape, regardless of which the backend actually returns. Other API wrappers that have similarly thin response envelopes should follow this pattern rather than pushing the shape difference up to the consumer.

## Iteration: Tuesday, May 5, 2026 11:55

**Story:** US-FE-SETTINGS-ZONES-01 - Pricing zone dropdown shows utility + region labels

**Status:** Completed

**Files Changed:**
- src/data/pricing-zones.ts (new — PRICING_ZONE_LABELS map + pricingZoneOptionLabel/pricingZoneFullLabel helpers; zones 1=SDG&E/San Diego, 2=ConEd/NYC, 3–8=TBD with TODO pointer to API repo's pricing.py)
- src/panel/hungry-machines-panel.ts (PRICING_ZONES now data-driven from PRICING_ZONE_LABELS; zone <option> rendered via pricingZoneOptionLabel with ?selected= per option; new <p class="zone-hint"> below the select reactively shows pricingZoneFullLabel(_zoneDraft); .zone-hint CSS rule)
- tests/pricing-zones.test.ts (new — four cases: zone-1 SDG&E regex, length === 8 with shape assertion, option-label format, full-label omits zone number)
- tests/settings.test.ts (new case asserting 8 options, zone-1 text matches /SDG.?E.*San Diego/, and .zone-hint reactively switches when the select changes)

**Learnings:**
- Both helper functions (`pricingZoneOptionLabel`, `pricingZoneFullLabel`) accept `number` rather than the strict `PricingZone` literal union, with a fallback to TBD. That lets the panel pass `_zoneDraft` (typed `number`) without casting and keeps a backend-introduced zone-9 from crashing the UI before the data file is updated.
- Tests should not assume `SAMPLE_USER.pricing_location` — it's 3, which means the initial `.zone-hint` shows "Provider TBD" not "SDG&E". Drove the new test off the *change* (zone → 1, then 2) rather than the initial render to keep it independent of the fixture's preset zone.
- Driving `PRICING_ZONES` off `Object.keys(PRICING_ZONE_LABELS).map(Number).sort()` (instead of the old `[1, 2, 3, 4, 5, 6, 7, 8] as const`) means the panel automatically picks up additions to the data file without a second edit. The cost is losing the literal-tuple type narrowing, which the option label helpers' `number` signature absorbs.

**Learnings:**
- Lit's HTML parser drops fragment-level `${TemplateResult}` slots, shifting later bindings by one position. The bug surfaces as bizarre output (a button's text rendering its own @click handler). Wrapping the interpolation in a host element (`<div>${X}</div>`) is the workaround. Documented in AGENTS.md and the codebase patterns block above.
- The `appliances.create()` API returns just `{appliance_id}` per the existing typed wrapper, but the story's contract called for `Promise<Appliance>`. Solved by normalizing on the way in: the wrapper accepts either shape from the server and synthesizes a complete Appliance from the request body when only the id comes back. This keeps the consumer side type-clean without needing a follow-up GET.
- The verification grep `grep -c 'appliancesApi.create\|appliances.create'` enforces the import style. Used `import * as appliancesApi from '../api/appliances.js'` so the call site reads as `appliancesApi.create(...)` rather than the more typical `import { create as createAppliance }` pattern. Worth keeping consistent across new code that touches this API.
