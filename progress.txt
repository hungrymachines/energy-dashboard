# RALPH Progress Log

## Codebase Patterns

- **EntityMap is panel-only.** `src/store.ts` exposes `getEntityMap` / `setEntityMap` for the panel's Settings tab. The two cards (`hm-thermostat-card`, `hm-savings-card`) take their entity wiring from Lovelace card config (`HmThermostatCardConfig.entities` / `HmSavingsCardConfig.entities`), NOT from the panel-level EntityMap. Don't conflate them when refactoring entity selection.
- **Legacy entity-map migration runs on read.** `getEntityMap()` filters out legacy keys (`indoor_temp`, `outdoor_temp`, `power`) and rewrites the localStorage entry when any legacy key is present. Future shape changes should follow the same one-shot read-side migration pattern rather than versioning.
- **Settings entity selects are domain-filtered.** Panel filters `Object.keys(hass.states)` by `id.startsWith('climate.')` for the climate field and `'weather.'` for the weather field. Tests must include at least one entity in each target domain plus a non-matching one (e.g. `light.kitchen`) to exercise filter behavior.
- **Test fixtures need attributes for new shape.** `climate.*` entities expose `current_temperature` and `temperature` (target) under `attributes`; `weather.*` exposes `temperature` (current outdoor) under `attributes`. Tests that simulate hass states should include those attributes so future card UI changes can read them without test churn.
- **Settings persistence is draft-then-save.** Entity map and pricing zone live in two layers: `_entityMap`/`_auth.user.pricing_location` are the saved baselines; `_entityMapDraft`/`_zoneDraft` are the in-memory drafts the form binds to. Change handlers only mutate drafts. The Save button is the single persistence path (calls `setEntityMap` and `patchMe`); Reset snaps drafts back to baselines; both buttons disable when `_isDirty()` is false.
- **`<select>` value binding for happy-dom remounts.** Lit's `.value=${selected}` does not reliably reflect on a select after a remount in happy-dom. Use `?selected=${id === selected}` per `<option>` (combined with the `.value` binding for the change-event path) so case-E-style "remount restores the saved value" tests pass without requiring real-browser semantics.
- **Display labels for backend lookups belong in `src/data/`.** When the backend exposes integer keys (pricing zones, appliance types) and the UI needs human-readable labels, drop the mapping in `src/data/<topic>.ts` as a typed `Record<KeyUnion, Label>` plus pure label-builder helpers. Keep helpers tolerant of out-of-range keys (fallback to a TBD label) so callers don't have to clamp. Drive `<option>` lists off `Object.keys(MAP).map(Number).sort()` instead of a hardcoded tuple — adding/removing a zone becomes a one-line data edit.
- **Wrap fragment-level `${TemplateResult}` interpolations in a host element.** When a Lit template has `${someTemplateResult}` at fragment level between sibling elements (e.g. `</label>\n      ${typeFields}\n      <div>`), the parser under happy-dom silently drops the binding slot. All bindings AFTER it shift by one position, leading to surreal output like a button's text rendering its own `@click` handler source. Wrap the slot in `<div>${typeFields}</div>` (or any host element) and the parser handles it correctly. This was a multi-hour debug rabbit hole on US-FE-APPLIANCES-01; future stories that compose multi-section templates should default to wrapping each section in its own element.
- **API `create` shape is normalize-on-the-way-in.** `appliancesApi.create()` accepts either a full `Appliance` body or a thin `{appliance_id}` envelope from the server and synthesizes an `Appliance`-shaped object from the input + response. Callers always get the same shape, regardless of which the backend actually returns. Other API wrappers that have similarly thin response envelopes should follow this pattern rather than pushing the shape difference up to the consumer.
- **Python tests use sys.modules stubs, not a real HA install.** RALPH iterations run without `homeassistant` / `aiohttp` / `voluptuous` available on PATH. `conftest.py` at the project root pre-installs minimal stubs for those modules (and the helper subpackages we use — `homeassistant.helpers.aiohttp_client`, `helpers.event`, `helpers.selector`, `components.frontend`, `components.http`) into `sys.modules` so `custom_components/hungry_machines/__init__.py` can be imported during test collection. Tests then `patch.object` the relevant pieces — typically `auth.aiohttp_client.async_get_clientsession` and `auth.refresh` / `auth.login` — to drive specific scenarios. Don't try to install real HA in the test loop; the stubs are sufficient and far faster.
- **Credentials live in entry.data; password is never persisted.** `config_flow.async_step_user` calls `auth.login(hass, email, password)`, then stores ONLY `{email, access_token, refresh_token, expires_at, climate_entity}` in entry.data. Reauth re-prompts for password only and replaces tokens via `hass.config_entries.async_update_entry`. The grep guard for password in `__init__.py / scheduler.py` (count == 0) plus `auth.py` (count > 0 only as a function param + JSON body, never an entry.data write) is the load-bearing invariant. Future config-flow steps must follow the same shape — no PASSWORD in any `data=` kwarg passed to `async_create_entry` / `async_update_entry`.
- **`/api/v1/schedules` schedule arrays are nested under `appliance.schedule.*`.** The contract is `{appliances: [{appliance_type, schedule: {intervals, high_temps, low_temps}, savings_pct, source, ...}]}`. v0.3.0–0.3.3 silently bug: `scheduler.py` read `hvac.get('high_temps')` (one level too high), so the cache was always empty and `apply_current_slot`'s slot-bounds check skipped every apply — no setpoint ever moved. Fixed in v0.3.4 by destructuring `sched = hvac.get('schedule') or {}` first. When mocking the schedules endpoint in tests, build the body with `schedule: {high_temps, low_temps, intervals}` nested on each appliance entry; legacy flat-shape bodies now correctly produce empty cache arrays (regression-marker test in `test_scheduler.py::test_fetch_schedule_legacy_flat_shape_yields_empty_cache`).
- **HVAC state must be one of HEAT/COOL/OFF/FAN.** The API's `SensorReading.hvac_state` validator (hungry-machines-api/app/routes/readings.py) hard-rejects any other value with a 422. HA's `climate.*` entities expose `state.state` like `'cool'`, `'heat'`, `'auto'`, `'heat_cool'`, etc. The integration's `readings.py` upper-cases then whitelists: anything not in `{HEAT, COOL, OFF, FAN}` falls back to `OFF`. Important: this includes `'auto'` and `'heat_cool'` — both map to `OFF` rather than guessing. A future story that wants finer grain (e.g. mapping `'heat_cool'` to `'OFF'` when idle vs `'COOL'` when actively cooling) would need to read the `hvac_action` attribute, not just `state.state`. Tests in `test_readings.py::test_push_current_reading_unknown_state_falls_back_to_off` lock the current behavior.
- **`hass.data[DOMAIN]` shape, expanded.** Now three keys: `_frontend_registered` (bool, set once per HA process), `schedule` (cached HVAC schedule across the single entry), `readings_unsub` (unsub for the 5-min readings poller — stored at the domain level, NOT in the per-entry `unsub` list, to avoid double-tear-down), plus `<entry.entry_id>` -> `{'unsub': [callbacks]}` for `async_track_time_change` subscriptions. `async_unload_entry` pops `readings_unsub` and calls it explicitly; the per-entry `unsub` list is iterated separately. New time-driven background work that's logically per-domain (not per-entry) should follow the same domain-level pattern.

A finer mapping (e.g. heat_cool + idle → OFF, heat_cool + cooling → COOL) would need `state.attributes['hvac_action']`, not state.state — out of scope here, locked by `test_push_current_reading_unknown_state_falls_back_to_off`.
- Storing the readings unsub at the domain level (`hass.data[DOMAIN]['readings_unsub']`) rather than appending to the per-entry `unsub` list prevents double-tear-down and matches the AC's literal wording. Per-entry unsubs (the 5:05 schedule refresh and the slot-apply track-time-change) keep their list-based pattern; domain-wide background work uses the explicit key. Documented in the codebase patterns block above.
- The test pattern reuses the now-canonical `_mock_response` / `_session_with_post` helpers from test_auth.py and test_scheduler.py (renamed inline in test_readings.py for module locality). The `_make_state` helper builds a MagicMock with `.state` and `.attributes` keys mirroring HA's State object — set attrs only when non-None so the test's `state.attributes.get('current_temperature')` returns None correctly when the user-facing scenario is "current_temperature missing".

The shared surface is just "get token, set Bearer, log on 401" — about 4 lines either way. Extracting would add a helper module with two divergent callers and net zero. The criterion explicitly said "if it'll cut code duplication" + "do not over-engineer", so kept the inline patterns. If a third API caller lands later this becomes worth re-evaluating.
